# Contributing

## Branching

- Используйте ветки `feature/<name>`, `fix/<name>`, `audit/<name>`, `chore/<name>`.
- Не смешивайте в одном PR миграции, крупный рефакторинг и новые фичи без необходимости.

## Setup

- Для локальной разработки сейчас используется `.env.dev`.
- Для stage/prod нельзя полагаться только на содержимое `.env.stage`/`.env.prod`; профиль должен задаваться внешней переменной `ENV`.
- Секреты не коммитятся в репозиторий. В git должен оставаться только `.env.example`.

## Quality Gates

- Перед PR прогоняйте `pytest -q`.
- Для изменений схемы БД обязательно прогоняйте `alembic upgrade head` на чистой БД.
- Если меняется публичный endpoint, обновляйте schema tests и документацию.

## Database Rules

- Не добавляйте `Base.metadata.create_all()` в runtime-путь приложения.
- Любые изменения схемы проводятся через Alembic.
- Миграция должна применяться на пустой БД и на актуальном `head`.

## API Rules

- Каждый публичный endpoint должен иметь явные Request/Response схемы.
- Не возвращайте “сырые” ORM-объекты или ad-hoc словари там, где endpoint публичный.
- Ошибки должны быть единообразными и покрыты тестами.

## Security Rules

- Секреты берутся только из окружения.
- JWT signing secret не должен быть захардкожен.
- Write-endpoints защищаются авторизацией.
- Логи не должны содержать `DATABASE_URL`, токены, полные prompt/raw-output с PII.

## Testing Rules

- Для каждого bugfix добавляйте как минимум один regression test.
- Для каждого критичного сценария пилота должен быть API или integration test.
- Внешние LLM и внешние источники в тестах должны мокаться.

## Review Expectations

- PR должен содержать краткое описание изменения, риск, способ проверки.
- Если меняется контракт API, приложите пример запроса/ответа.
- Если меняется БД, укажите миграцию и план отката.
